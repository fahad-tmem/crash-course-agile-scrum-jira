Part 1: The Core Concepts

Before diving into Jira, it's crucial to understand the "why". This section breaks down the foundational ideas of Agile and the Scrum framework.

Agile: The Mindset

Agile is not a strict process, but a philosophy for building products. It values adaptability and collaboration over rigid planning.

  • ✔Iterative Development: Build and release in small, frequent cycles (sprints).
  • ✔Collaboration: Daily interaction between the team and stakeholders.
  • ✔Responding to Change: Welcome changing requirements, even late in development.
  • ✔Focus on Value: Prioritize delivering working, valuable software frequently.

Scrum: The Framework

Scrum is a practical framework for implementing the Agile mindset. It provides a structured set of roles and events to manage work.

Key Roles:
Product Owner

Decides *what* to build by managing the Product Backlog.

Scrum Master

Facilitates the process and removes blockers for the team.

Development Team

Decides *how* to build the product and does the work.

Key Events (Ceremonies):
The Sprint

A 2-4 week timebox where a shippable increment of work is created.

Sprint Planning

Team selects work from the backlog for the upcoming Sprint.

Daily Stand-up

A 15-min daily sync to discuss progress and impediments.

Sprint Review

Team demonstrates completed work to stakeholders.

Sprint Retrospective

Team reflects on the sprint to identify improvements.

Part 3: Writing Great Issues

Clear, well-written issues prevent confusion and save time. Use these templates as a guide for creating effective tickets in Jira.

Template: User Story

Summary (Title)

As a <user type>, I want <to perform an action>, so that <I can achieve a goal>.

Example: As a registered user, I want to reset my password, so that I can regain access to my account.

Acceptance Criteria (AC)

This is a critical checklist that defines "Done". Use the Given/When/Then format for clarity.

Scenario: User enters a valid email
Given I am on the "Forgot Password" page
When I enter my registered email address and click "Submit"
Then I should see a confirmation message.

Template: Bug Report

Summary (Title)

[Area of App] - Brief description of the problem.

Example: [Checkout Page] - "Complete Purchase" button is disabled after entering valid card info.

Steps to Reproduce

  1. Navigate to the shopping cart.
  2. Click "Checkout".
  3. Enter valid shipping and payment info.
  4. Observe the "Complete Purchase" button.

Expected vs. Actual

Expected: Button becomes active.

Actual: Button remains disabled.

Environment

Browser: Chrome v105, OS: Windows 11

Template: Task

Summary (Title)

Be direct and action-oriented.

Example: Upgrade the production server to version 2.5

Description

  • Why: Explain the reason for the task.
  • What: Provide a clear list of steps or requirements.
  • Definition of Done: How will we know this is complete? (e.g., Server is running v2.5 with no errors for 1 hour).

Template: Epic

Summary (Title)

A clear, high-level name for a large feature.

Example: User Profile Management

Description

  • Goal: Briefly explain the business objective. What problem are we solving?
  • Scope: List the high-level features or User Stories that will be part of this Epic.
  • Out of Scope (Optional): What are we explicitly not doing?

Part 4: The Basic Workflow

Our Jira board provides a visual overview of our sprint's progress. The goal is to move all work from "To Do" to "Done" within the sprint.

To Do

Work planned for the sprint but not yet started.

In Progress

An issue that a team member is actively working on.

Done

Work that is 100% complete, tested, and meets all requirements.
